# 🏺 맛담 (MatDam) — 마스터 프로젝트 기획서

> **한식 중심의 글로벌 커뮤니티 레시피 플랫폼**
> *"세상의 모든 맛을 담고 잇다"*
>
> 최종 업데이트: 2026-02-13

---

## 1. 프로젝트 개요 및 비전

### 1.1 핵심 컨셉

"맛을 담다" — 단순한 정보 전달을 넘어, 하나의 레시피가 전 세계로 퍼져나가며 어떻게 현지화되고 진화(Remix)하는지 기록하는 **레시피 버전 관리 플랫폼**.

### 1.2 시장 포지셔닝

**기존 한계:** Maangchi 등 기존 강자들은 1인 블로그/유튜브 기반의 일방향 콘텐츠에 치중. 사용자가 자기만의 버전을 공유하고 소통할 플랫폼이 부재.

**맛담의 차별점:**
> **Maangchi의 신뢰도 + 쿡패드의 커뮤니티 + GitHub의 버전 관리**

### 1.3 타겟 사용자

- **Primary:** 한식에 관심있는 외국인 (영어권 우선)
- **Secondary:** 해외 거주 한국인
- 플랫폼 언어: 영어 기본, 한국어 병행 (next-intl 기반 다국어 라우팅)

### 1.4 핵심 철학

- 플랫폼이 콘텐츠를 만드는 게 아니라, **사람들이 실험하고 공유하는 판을 깔아주는 것**
- 검증되지 않은 정보를 플랫폼 이름으로 제공하지 않음
- 사용자 경험이 쌓여서 자연스럽게 검증된 정보가 되는 구조

### 1.5 글로벌 확장 비전

한식에서 시작하되, 국가를 추가하는 방식으로 확장 예정. '맛을 담다'라는 의미는 한식에 국한되지 않고 모든 음식에 적용 가능한 보편적 가치.

| Phase | 집중 분야 | 주요 내용 |
|-------|----------|----------|
| Phase 1 (Origin) | 한식 (K-Food) | 정통 레시피 + 서구권 퓨전 리믹스 중심 |
| Phase 2 (Expansion) | 인접 국가 및 인기 퀴진 | 일식, 중식, 동남아 확장. 식재료 교차 추천 강화 |
| Phase 3 (Global) | 대륙별 퀴진 | 유럽, 남미 등 국가별 섹션 분리 및 현지 인플루언서 협업 |

국가 확장 시 Fork & Remix 시스템이 진가를 발휘 — 국가 간 식재료와 조리법이 섞이는 과정을 시각화하고 데이터화하는 것이 플랫폼의 핵심 자산.

## 3. 핵심 시스템 1: 다국어 식재료 매핑 엔진 (Global Ingredient Engine) ⭐

플랫폼의 '두뇌'이며, 글로벌 확장과 지능형 추천의 핵심.

### 3.1 Universal Ingredient ID

모든 식재료는 고유 ID를 가지며, 다국어 매핑과 단위 변환의 기반.

```
ing_gochujang_001
├── ko: 고추장
├── en: Gochujang (Korean red chili paste)
├── fr: Pâte de piment coréenne
├── ja: コチュジャン
└── local_names: { "us": "Korean hot pepper paste", ... }
```

### 3.2 단위 변환

- Metric (g, ml) ↔ Imperial (oz, cup) 실시간 변환
- 레시피 보기 화면에서 원클릭 토글
- 사용자 국가(IP 기반 Geo-IP)에 따라 기본 단위 자동 설정
- 단위 변환 로직은 유틸리티 함수로 공통화하여 국가 확장 시 재활용

### 3.3 Ingredient Glossary (재료 사전)

- 한식 재료 사진 + 다국어 설명
- 비교 정보: "된장은 miso와 비슷하지만 발효 방식이 다르다"
- 레시피에서 재료 클릭 시 glossary로 바로 연결
- 구매 안내: "주로 Asian grocery stores에서 구매 가능"
- 향후 커뮤니티가 확장하면 '글로벌 식재료 위키' 형태로 발전
- **고등급 인증 유저에게 편집 권한 부여** (위키 스타일 운영)

### 3.4 현지 대체재 매핑 (Substitutes)

- **Visual Picker:** 특정 국가에서 구하기 힘든 재료(예: 깻잎)를 현지 대안(예: 시소)으로 시각적 제안
- **성공률 데이터:** 사용자들이 리믹스 과정에서 남긴 대체재 활용 후기를 데이터화하여 신뢰도 측정
- **주의:** 플랫폼이 정답을 주는 게 아님 → 사용자 경험 축적 구조
- **(향후) Local Grocery Finder:** 사용자 국가 정보 기반으로 해당 재료를 살 수 있는 현지 마켓(H-Mart 등)이나 온라인몰 링크 연결

### 3.5 Dietary Filter

- Vegan / Halal / Gluten-free / Nut-free 등
- 한식은 젓갈(fish sauce), 간장(soy=gluten) 등이 기본이라 차별화 포인트

**필터 UX 전략 (소프트/하드 분리):**
- **기본값:** 라벨 표시(Soft Filter) — 모든 레시피 보이되, 적합 여부를 마크로 표시. 부적합 레시피는 흐리게(Dimmed) 처리하거나 경고 문구 표시. 부적합 레시피도 "이 레시피의 [비건/할랄] 리믹스 버전이 있어요!" 연결 가능 → 리믹스 문화 활성화
- **설정 옵션:** 엄격한 필터링(Hard Filter) — 부적합 레시피 완전 숨김. 사용자가 직접 토글

## 4. 핵심 시스템 2: Fork & Remix 생태계 (The Evolution of Taste)

### 4.1 기본 구조

- 쿡패드 츠쿠레포 + GitHub fork 개념
- 원본 한식 레시피에서 자기만의 퓨전 버전을 "remix"로 올림
- **리믹스 자체는 항상 자유** — 누구나 아무 레시피를 리믹스 가능

### 4.2 계층 구조 및 승인권 (Hierarchy & Approval)

**레시피 트리(Recipe Tree):** 오리지널(Root) → 리믹스(Branch) → 리믹스의 리믹스

```
🌳 오리지널 떡볶이 (Root)
├── 라볶이 (1차 리믹스) ← 오리지널 제작자가 승인
│   ├── 치즈 라볶이
│   └── 마라 치즈 라볶이 ← 라볶이 제작자가 승인
├── 짜장 떡볶이
└── 로제 떡볶이 ⭐ Master's Choice
```

**직계 부모 승인권:**
- "마라 치즈 라볶이"의 공식 리믹스 여부는 **직계 부모인 '라볶이' 제작자**가 결정
- 오리지널(Root) 제작자가 아닌 직계 부모가 결정하는 이유:
  - 확장성: 한 사람에게 병목 방지
  - 전문성: 중간 단계 제작자가 해당 영역을 더 잘 앎

**뿌리 제작자(Root)의 명예:**
- 가문 전체에서 뛰어난 레시피에 **'Master's Choice'** 배지 부여 권한
- 트리 전체의 파생 현황 통계 확인 가능

**핵심 원칙:** 승인권은 "공식 리믹스로 추천할지"에 한정. 리믹스 생성 자체를 막는 것은 아님.

### 4.3 저작권 보호 및 예우

**Inspired by 시스템:**
- 타인의 레시피를 참고했을 때 출처를 자동 명시
- 레시피 입력 폼에 'Inspired by' / 'Reference' 필드 제공
- 플랫폼 내 리믹스: "이 레시피는 [원본]의 remix입니다" 자동 링크
- 외부 참고: "Inspired by: [Maangchi's YouTube Channel]" 수동 입력

**Safe Harbor / DMCA 대응:**
- 미국 DMCA + 한국 저작권법(OSP 의무) 이중 준수
- 신고 접수 시 즉시 삭제(Notice and Takedown) 프로세스
- 사용자 신고 시스템을 MVP 단계부터 포함

**이용약관(ToS) 핵심 조항:**
- 레시피 올릴 때 "다른 사용자가 이 레시피를 Remix하는 것을 허용한다" 동의 (Creative Commons 스타일)
- 콘텐츠 저작권은 사용자에게 있되, 플랫폼이 서비스 홍보/다국어 번역에 사용할 권한 보유
- 준거법: 대한민국 법 (약관에 명시)

**레시피 저작권 참고:**
- 재료 목록과 분량(수치)은 저작권 보호 대상이 아님 (사실/데이터 영역)
- "고추장 1스푼"을 "고추장: 1스푼"으로 구조화하는 것은 데이터 변환이므로 문제 없음
- 기능적 조리 순서("물이 끓으면 면을 넣으세요")도 창작성 인정 어려움
- **보호 대상:** 사진, 영상, 독창적 설명 텍스트(미사여구/에피소드/팁), 레시피 모음집
- 유튜브/방송 레시피의 재료+분량을 구조화 등록하는 것은 법적으로 문제 없음
- 단, 설명 텍스트는 본인의 말로 재구성 필수, 사진은 직접 촬영 필수
- **출처 표기:** 법적 의무는 아니지만 "○○의 레시피를 참고하여 만들었습니다" 식으로 밝히면 신뢰도 향상 + 마찰 방지

**국제 저작권 원칙:**
- 베른 협약(Berne Convention)에 따라 '보호가 요청되는 국가의 법' 적용 (보호국법 원칙)
- 서버가 한국에 있어도 서비스가 닿는 모든 국가의 저작권자로부터 소송 가능성 있음
- 대응: DMCA 수준의 '신고 및 즉시 삭제' 절차를 갖추는 것이 가장 안전

### 4.4 구현 로드맵

- **V1:** "이 레시피는 [원본]의 remix입니다" 링크로 심플하게
- **V2:** 트리 시각화, 승인권 시스템, Master's Choice 배지

## 5. 구조화된 리뷰 시스템

### 5.1 축별 평가

- **Taste** (맛)
- **Difficulty** (난이도)
- **Authenticity** (정통성)
- **Kid-friendly** (아이 적합도)

### 5.2 "I made this!" 후기

- 사진 필수 후기
- 리뷰 기반 레시피 신뢰도 스코어

### 5.3 사진 신뢰도 시스템 (4단계)

| 단계 | 검증 방법 | 목적 |
|------|----------|------|
| 1단계 (자동) | **pHash 검사** | 플랫폼 내 기존 사진과의 중복 여부 즉시 확인 |
| 2단계 (자동) | **EXIF 분석** | 원본 파일 여부 확인, 촬영 기기/날짜 기록 → **신뢰 점수 반영** (차단 X) |
| 3단계 (커뮤니티) | **"I made this!" 리뷰** | 다른 유저들이 실제 결과물 보고 남기는 리뷰 기반 검증 |
| 4단계 (사후) | **유저 신고** | 저작권 침해 의심 시 운영자에게 신고 |

**EXIF 주의사항:** 블로거가 자기 블로그에서 사진을 다운받아 올리면 EXIF가 삭제된 상태일 수 있음. 따라서 EXIF 없음 = 차단이 아니라 "일반 유저" 처리. EXIF 있음 = "직접 촬영 확률 높음" 인증 마크 부여.

**워터마크 자동 삽입 (선택적):** 사용자가 사진 업로드 시 '맛담' 로고 + 사용자 ID를 워터마크로 자동 삽입하는 옵션 제공 → 외부 무단 유출 심리적 방지

## 6. 사용자 기능 및 UX

### 6.1 쿠킹 모드 (Cooking Mode) ⭐

플랫폼 차별화의 핵심 기능. 만개의레시피, 해먹남녀에도 없음.

**기본 구조:**
- 일반 모드: 전체 레시피가 한 페이지에 (읽기/미리보기용)
- 쿠킹 모드 진입: 풀스크린 전환, 스텝별 카드 한 장씩 (포스트잇/카드 느낌)

**카드별 구성:**
- 그 스텝에서 필요한 재료만 하이라이트
- 해야 할 동작 한두 문장
- 타이머 필요 시 바로 시작 버튼
- 좌우 스와이프 또는 큰 "다음" 버튼

**디바이스별 반응형 레이아웃:**

| 화면 크기 | 레이아웃 | 비고 |
|-----------|----------|------|
| ~480px (폰) | 현재 스텝 풀카드 + 하단에 다음 스텝 첫 줄 peek | 위로 스와이프하면 전환 |
| 481~1024px (태블릿) | PPT 발표자 보기 스타일 — 상위 70~75% 현재 스텝, 하단 25~30% 다음 스텝 미리보기 | 다음 스텝은 흐리게 처리 |
| 1025px~ (데스크톱) | 좌우 분할 가능 | |

**기기 판별:** `window.innerWidth` 기반 반응형 (UA 파싱 비추). 기기를 맞추는 게 아니라 "화면이 충분히 넓은가?"를 보는 것.

**부가 기능:**
- Wake Lock API (`navigator.wakeLock`) — 화면 안 꺼지게
- 복수 타이머 관리 — 카드 넘겨도 이전 스텝 타이머 유지, 플로팅 타이머 표시
- 손 젖었을 때 대응 — 큰 버튼, 화면 아무 곳 탭으로 넘기기
- (향후) Web Speech API 음성 "다음"

### 6.2 스텝별 레시피 입력 구조

쿠킹 모드를 위해 입력 단계에서부터 스텝 분리 필요.

**데이터 구조:**
```json
{
  "recipe_id": "dubu-jorim-001",
  "title": "Dubu Jorim",
  "ingredients": [
    {"id": "ing1", "name": "Tofu", "amount": "1", "unit": "block"},
    {"id": "ing2", "name": "Vegetable oil", "amount": "2", "unit": "tbsp"}
  ],
  "steps": [
    {
      "order": 1,
      "description": "Slice tofu into 1cm thick pieces...",
      "ingredients_used": ["ing1"],
      "timer_seconds": null,
      "image_url": null
    },
    {
      "order": 2,
      "description": "Pan-fry on medium heat until golden...",
      "ingredients_used": ["ing2"],
      "timer_seconds": 180,
      "image_url": null
    }
  ]
}
```

**입력 방식:** 구조화 폼 기본 + 자동 파싱 보조
- 기본: 스텝별로 나눠서 입력하는 구조화된 폼
- 보조: "텍스트로 한번에 입력" → 자동 분리 후 구조화 폼에 채움
- (V2) AI 기반 레시피 자동 파싱 — 유튜브 URL이나 블로그 글을 넣으면 구조화된 JSON으로 변환
- (V2) AI Auto-Tagging — 레시피 본문을 분석해 Vegan, Gluten-free, Halal 등 필터를 자동 제안

### 6.3 지능형 추천 시스템

**식재료 연쇄 추천 (Ingredient Chaining) — 선제적 추천:**
- 떡볶이를 만들고 남은 양배추 3/4통을 버리지 않도록, 이어서 만들 수 있는 최적의 레시피를 선제적으로 제안
- 레시피 하단 또는 쿠킹 모드 완료 화면에 추천 카드 배치
- Universal Ingredient ID 기반 매칭
- 사용자의 Dietary Filter + 난이도 고려
- **현지 구매 단위 반영:** 한국은 양배추 1/4통 판매 가능하지만, 미국/유럽은 통째로 판매 → "통째로 샀을 때" 기준으로 남은 양 계산하면 외국인 UX 향상

**냉장고 털이 — 사후 검색형:**
- 보유 재료를 다국어 ID 기반으로 검색
- 사용자의 식단(비건, 할랄 등)에 맞는 최적의 리믹스 버전 노출

**연쇄 추천 vs 냉장고 털이:**
- 연쇄 추천 = Proactive: "이 요리 하면 이만큼 남을 테니, 다음엔 이걸 하세요"
- 냉장고 털이 = Reactive: "이미 남은 재료가 있는데 뭘 하지?"

### 6.4 Shopping List 생성

- 여러 레시피 골라서 "이번 주 장보기 리스트" 자동 생성
- 재료 겹치면 합산 (Universal Ingredient ID 활용)

### 6.5 Recipe Linter (품질 검수)

레시피 등록 전 자동 품질 체크:
- 필수 단위(Metric/Imperial) 누락 여부
- 모호한 설명 감지 (예: '적당히', '한 꼬집' → 구체적 수치 권장)
- 글로벌 유저가 이해하기 쉬운 표준화된 품질 유지

### 6.6 온보딩 플로우 (Dietary Preferences 수집)

| 단계 | 방식 | 내용 |
|------|------|------|
| Step 1 | 간편 가입 | 소셜 로그인(Google, Apple)으로 최소 정보만 |
| Step 2 | 선택적 온보딩 | "맞춤 한식을 찾아드릴까요?" 팝업으로 비건/할랄/난이도 체크 유도 |
| Step 3 | 프로필 설정 | 언제든 마이페이지에서 선호도 수정 가능 |

수집 항목: 식단 제한(Vegan, Halal, Gluten-free, Nut-free), 요리 숙련도(Beginner~Master), 선호 언어

### 6.7 게이미피케이션 (Gamification)

**글로벌 히트맵:**
- 메인 페이지에 각 국가별 레시피 등록 현황을 지도로 표시
- 참여 경쟁 유도 ("우리 나라 레시피가 아직 없네? 내가 올려야지!")
- 초기 데이터 부족 시: 숫자 0인 나라는 표시하지 않거나, 최신 리믹스 올라온 국가만 강조
- **UI 옵션들:**
  - 인터랙티브 히트맵: 레시피 많은 지역 진한 색 → 클릭 시 해당 국가 레시피 검색
  - Country Leaderboard: "Most Active Countries This Week" 상위 5~10개국 국기+순위
  - "Now Cooking In..." 실시간 알림: "A user in France just remixed Kimchi Stew!" 토스트/띠 형태
- **데이터 집계:** 매번 실시간 count 대신 일정 주기로 캐싱하여 메인에 표시 (DB 부하 방지)

**사용자 등급 및 인센티브:**

| 등급 | 조건 | 주요 권한 |
|------|------|----------|
| Beginner | 가입 | 레시피 등록, 리뷰 작성 |
| Intermediate | 인증된 사진 N개 + 리뷰 N개 | 검색 결과 우선 노출, 프로필 커스텀 |
| Master | 꾸준한 인증 콘텐츠 + 리믹스 활동 | **Glossary 편집권**, 리믹스 승인권, 분석 대시보드 |

**인센티브 체계:**
- 인증 사진 포함 레시피 → 검색 결과/메인 페이지 우선 노출
- "I made this!" 인증 리뷰 → 리뷰 영역 상단 고정
- 리믹스 통계 제공 — 자신의 레시피가 전 세계 몇 개국으로 뻗어나갔는지
- (향후) 포트폴리오 페이지 — 프로필을 개인 포트폴리오로 활용, 전용 URL 제공
- (향후) 공식 SNS 피처링 — 고퀄리티 인증 레시피를 TikTok/Instagram에서 홍보

**Kitchen Crew (그룹 리믹스):**
- 특정 테마(예: "뉴욕에서 즐기는 추석 음식")를 위해 여러 유저가 팀을 이뤄 하나의 Remix Tree를 완성하는 프로젝트형 메뉴
- Mentorship Badge: 초보의 질문에 성실히 답변하거나 리믹스를 도와준 숙련자에게 부여

## 8. 기술 설계 및 인프라

### 8.1 기술 스택

| 영역 | 기술 | 이유 |
|------|------|------|
| **Frontend** | Next.js (App Router) | 글로벌 SEO, 다국어 라우팅(next-intl), SSR/SSG |
| **Backend/DB** | Supabase (PostgreSQL) | 관계형 트리 구조(Recursive Query), jsonb로 유연한 Preferences, Auth/Storage 통합 |
| **Styling** | Tailwind CSS | 쿠킹 모드 등 기기별 복잡한 반응형 UI 빠른 구현 |
| **Search** | Typesense 또는 Meilisearch | 다국어 오타 교정 (Kimbap→Gimbap), 빠른 검색 |
| **Infrastructure** | Vercel | 글로벌 엣지 네트워크, Edge Functions, Image Optimization |
| **인증** | Supabase Auth | 소셜 로그인 (Google, Apple) |

**Supabase 유지 근거:**
- 단순히 무료라서가 아니라, PostgreSQL의 Recursive Query가 레시피 트리 추적에 강력
- Auth + Storage + Realtime 통합 환경으로 1인 개발 시 DevOps 부담 최소화
- 무료 티어로 MVP, 트래픽 증가 시 Pro로 전환
- 글로벌 엣지 네트워크를 통한 빠른 응답 (개인 NAS 대비 압도적 속도 차이)

**대체제 참고:** PlanetScale(MySQL) — 전 세계 수백만 동시 접속 수준의 수평 확장이 필요한 경우 고려. 단, 초기 개발 속도와 통합 환경은 Supabase가 유리.

**다국어/현지화 구현:**
- Next.js Internationalized Routing: `/en/recipe/bulgogi`, `/ko/recipe/bulgogi`
- 정적 텍스트(메뉴, 버튼): next-intl 처리
- 동적 데이터(레시피 본문): DB에서 jsonb 다국어 필드 또는 별도 번역 테이블
- 향후 커뮤니티 번역(Community Translation) 구조 고려

### 8.2 쿠킹 모드 기술 포인트

- 디바이스 분기: CSS media query / `window.innerWidth`
- 화면 유지: Wake Lock API
- 타이머: 복수 타이머 관리, 카드 전환 시에도 유지
- 터치 최적화: 큰 터치 타겟, 스와이프 제스처
- 터치 감지: `'ontouchstart' in window` (보조 수단)

### 8.3 레시피 DB 스키마 (초안)

```
Recipe
├── recipe_id (PK)
├── title (다국어 jsonb)
├── author_id (FK → User)
├── parent_recipe_id (FK → Recipe, nullable) ← 직계 부모 Remix 관계
├── root_recipe_id (FK → Recipe, nullable) ← 트리 최상위
├── ingredients[] ← Universal Ingredient ID 참조
│   ├── ingredient_id
│   ├── amount
│   └── unit
├── steps[]
│   ├── order
│   ├── description
│   ├── ingredients_used[]
│   ├── timer_seconds
│   └── image_url
├── dietary_tags[] (vegan, halal, gluten-free...)
├── difficulty_level (beginner/intermediate/master)
├── inspired_by (text, nullable) ← 외부 참고 출처
├── is_approved_remix (boolean) ← 직계 부모 승인
├── master_choice (boolean) ← Root 제작자 배지
└── created_at / updated_at

User Profile (Supabase auth.users 연동)
├── user_id (PK)
├── display_name
├── country
├── tier (beginner/intermediate/master)
├── trust_score (float)
├── preferences (jsonb) ← { dietary: [...], skill_level: "...", preferred_unit: "..." }
└── created_at
```

## 9. 마케팅 및 운영 전략

### 9.1 시드 콘텐츠 전략

- 10~20개 고퀄리티 오리지널 한식 레시피 직접 제작

- **단순 정보 전달이 아니라 "이 플랫폼은 이렇게 노는 곳이다" 본보기:**
  - 원본: 정통 된장찌개
  - 리믹스 1: (미국 현지) 주키니와 파마산 치즈를 넣은 된장찌개
  - 리믹스 2: (비건) 멸치 육수 대신 채수를 사용한 된장찌개

### 9.2 Phase 1: 씨뿌리기 (런칭 전~직후)

**Reddit 게릴라 마케팅 (초반 가장 현실적):**
- r/KoreanFood, r/cooking 등에서 먼저 가치를 줌
- "현지 재료로 구현한 한식 레시피 PDF/이미지"를 먼저 공유
- 반응이 좋을 때 "더 많은 변형 버전을 여기서 볼 수 있다" 식으로 유도
- **절대 광고처럼 하면 안 됨** — Value-First 전략

### 9.3 Phase 2: 콘텐츠 엔진 (1~3개월)

- TikTok/Reels 30~60초 영상 → "Full recipe on [사이트]"
- K-drama/K-pop 연계: 자유 태그로 처리, 별도 시스템 불필요
- 해시태그: #koreanfood #easykoreancooking #kfood

### 9.4 Phase 3: 커뮤니티 부스팅 (3~6개월)

- 주간 챌린지: #ShowMeYourBibimbap
- non-Korean 마이크로 인플루언서 협업

### 9.5 SEO 전략

- 레시피 플랫폼은 구글 검색 유입이 생명
- Next.js SSR로 레시피 페이지를 검색 엔진에 최적화
- 각 언어권별 "Easy [Cuisine] Recipe" 키워드 선점
- 인증된 사진/원본 데이터가 확실한 레시피에 SEO 가산점

## 13. 현실 체크

1. **혼자서 모든 걸 하기 힘듦** — 사이트 개발 + 시드 콘텐츠 + 쇼폼 영상 + 커뮤니티 관리. 우선순위 필요.
2. **닭과 달걀** — 사람이 없으면 콘텐츠 X, 콘텐츠 없으면 사람 X. 시드 콘텐츠가 핵심. 시드는 "이 플랫폼은 이렇게 노는 곳이다"를 보여주는 본보기여야 함.
3. **영어 콘텐츠 퀄리티** — 타겟이 외국인이면 영어가 자연스러워야 함.
4. **수익화까지 시간** — 광고 수익은 트래픽 확보 후. 단기 수익 기대하기 어려움.
5. **SEO 장기전** — 꾸준한 콘텐츠 업로드와 키워드 최적화 필요.
6. **다국어 재료 엔진 범위 관리** — MVP에서는 주요 한식 재료 50~100개로 시작, 커뮤니티가 확장하는 구조가 현실적.
7. **승인권 시스템 복잡도** — 직계 부모 승인권, Master's Choice 등은 V2로 미루는 게 맞음.
8. **글로벌 히트맵 초기 공허함** — 데이터 적을 때 지도가 텅 비어 보이는 문제. 숫자 0인 나라 숨기기 등 전략 필요.
9. **문화적 맥락 반영** — 단순 번역이 아닌 해당 국가의 식사 예절, 주방 환경(오븐 사용 여부 등) 고려 필요.
