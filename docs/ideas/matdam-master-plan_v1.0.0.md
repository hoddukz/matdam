# 🏺 맛담 (MatDam) — 마스터 프로젝트 기획서

> **한식 중심의 글로벌 커뮤니티 레시피 플랫폼**
> *"세상의 모든 맛을 담고 잇다"*
>
> 최종 업데이트: 2026-02-13

---

## 1. 프로젝트 개요 및 비전

### 1.1 핵심 컨셉

"맛을 담다" — 단순한 정보 전달을 넘어, 하나의 레시피가 전 세계로 퍼져나가며 어떻게 현지화되고 진화(Remix)하는지 기록하는 **레시피 버전 관리 플랫폼**.

### 1.2 시장 포지셔닝

**기존 한계:** Maangchi 등 기존 강자들은 1인 블로그/유튜브 기반의 일방향 콘텐츠에 치중. 사용자가 자기만의 버전을 공유하고 소통할 플랫폼이 부재.

**맛담의 차별점:**
> **Maangchi의 신뢰도 + 쿡패드의 커뮤니티 + GitHub의 버전 관리**

### 1.3 타겟 사용자

- **Primary:** 한식에 관심있는 외국인 (영어권 우선)
- **Secondary:** 해외 거주 한국인
- 플랫폼 언어: 영어 기본, 한국어 병행 (next-intl 기반 다국어 라우팅)

### 1.4 핵심 철학

- 플랫폼이 콘텐츠를 만드는 게 아니라, **사람들이 실험하고 공유하는 판을 깔아주는 것**
- 검증되지 않은 정보를 플랫폼 이름으로 제공하지 않음
- 사용자 경험이 쌓여서 자연스럽게 검증된 정보가 되는 구조

### 1.5 글로벌 확장 비전

한식에서 시작하되, 국가를 추가하는 방식으로 확장 예정. '맛을 담다'라는 의미는 한식에 국한되지 않고 모든 음식에 적용 가능한 보편적 가치.

| Phase | 집중 분야 | 주요 내용 |
|-------|----------|----------|
| Phase 1 (Origin) | 한식 (K-Food) | 정통 레시피 + 서구권 퓨전 리믹스 중심 |
| Phase 2 (Expansion) | 인접 국가 및 인기 퀴진 | 일식, 중식, 동남아 확장. 식재료 교차 추천 강화 |
| Phase 3 (Global) | 대륙별 퀴진 | 유럽, 남미 등 국가별 섹션 분리 및 현지 인플루언서 협업 |

국가 확장 시 Fork & Remix 시스템이 진가를 발휘 — 국가 간 식재료와 조리법이 섞이는 과정을 시각화하고 데이터화하는 것이 플랫폼의 핵심 자산.

### 1.6 영어 UX 1순위 원칙

Primary 타겟이 "한식에 관심 있는 외국인"이므로, **개발 첫날부터 영어를 기본 언어로 설계.** 한국어는 두 번째 언어로 추가.

- URL 구조: `/en/recipe/tteokbokki` (기본) / `/ko/recipe/tteokbokki` (보조)
- UI/UX 검증 기준: **"외국인이 이 화면을 처음 봤을 때 3초 안에 이해하는가?"**
- 시드 콘텐츠: 영어 우선 작성 → 한국어 병행
- next-intl 기본 locale: `en`
- 한국어 기준으로 설계 후 번역하면 런칭 후 전부 뜯어고쳐야 함 — 이건 번역 순서가 아니라 **사고방식의 전환**

## 2. 시장 분석

### 2.1 글로벌 K-Food 트렌드

- 2025년 한국 농식품 수출 전년 대비 ~10% 증가
- 미국 시장 Q2 2025 전년 대비 28.6% 성장
- K-pop/K-drama가 해외 한식 인기를 견인
- 불고기 타코, 비빔밥 피자, 김치 토핑 등 퓨전 요리 전 세계 확산
- TikTok #koreanfood 태그 아래 영상 조회수 폭발적

### 2.2 기존 플레이어 분석

| 플랫폼 | 형태 | 강점 | 약점 |
|--------|------|------|------|
| **Maangchi** | 1인 블로그+유튜브 | 영어권 한식 #1, 충성 팬층 | 일방향, 사용자 참여 불가 |
| **My Korean Kitchen** | 1인 블로그 | 접근성 좋은 설명 | 커뮤니티 없음 |
| **Korean Bapsang** | 1인 블로그 | 정통 가정식 포커스 | 커뮤니티 없음, 퓨전 부재 |
| **FutureDish** | 부부 유튜브+블로그 | 400+ 레시피, 영상 튜토리얼 | 일방향, 사용자 업로드 불가 |
| **만개의레시피** | 대형 플랫폼 | 2,400+ 레시피, AI 기능 | 한국어 전용, 외국인 대상 아님 |

### 2.3 핵심 발견: 빈 자리

**"한식 중심 + 사용자 참여형 + 커뮤니티" 글로벌 플랫폼이 아예 없음**

### 2.4 SWOT 분석

| 강점 (Strengths) | 약점 (Weaknesses) |
|------------------|-------------------|
| '버전 관리' 개념의 독창적 UX | 초기 시드 콘텐츠 확보 부담 |
| 철저한 사용자 중심 (단위 변환, 필터) | 1인 개발/운영 시 마케팅 리소스 부족 |

| 기회 (Opportunities) | 위협 (Threats) |
|---------------------|----------------|
| K-드라마/K-팝 결합 마케팅 용이 | 거대 플랫폼(만개의레시피 등)의 글로벌 진출 |
| 식재료 이커머스 제휴 확장성 | 생성형 AI 기반 레시피 서비스의 급성장 |

**AI 레시피 서비스 대응 전략:**
ChatGPT/Gemini에게 "한식 레시피 알려줘"하면 즉시 답을 주는 시대. 맛담이 제공해야 하는 건 **AI가 줄 수 없는 것 = 실제 사람들의 검증된 결과물.**
- "I made this!" 인증 사진과 리뷰를 레시피 카드에서 **가장 눈에 띄는 위치**에 배치
- 리뷰 수, 리믹스 수, 인증 사진 수를 신뢰 지표로 전면에 노출
- AI 레시피 = 이론적 / 맛담 레시피 = 실전 검증이라는 포지셔닝
- 초기 시드 콘텐츠부터 반드시 직접 요리한 사진 포함 → "진짜 만들어본 레시피" 강조

## 3. 핵심 시스템 1: 다국어 식재료 매핑 엔진 (Global Ingredient Engine) ⭐

플랫폼의 '두뇌'이며, 글로벌 확장과 지능형 추천의 핵심.

### 3.1 Universal Ingredient ID

모든 식재료는 고유 ID를 가지며, 다국어 매핑과 단위 변환의 기반.

```
ing_gochujang_001
├── ko: 고추장
├── en: Gochujang (Korean red chili paste)
├── fr: Pâte de piment coréenne
├── ja: コチュジャン
└── local_names: { "us": "Korean hot pepper paste", ... }
```

### 3.2 단위 변환

- Metric (g, ml) ↔ Imperial (oz, cup) 실시간 변환
- 레시피 보기 화면에서 원클릭 토글
- 사용자 국가(IP 기반 Geo-IP)에 따라 기본 단위 자동 설정
- 단위 변환 로직은 유틸리티 함수로 공통화하여 국가 확장 시 재활용

### 3.3 Ingredient Glossary (재료 사전)

- 한식 재료 사진 + 다국어 설명
- 비교 정보: "된장은 miso와 비슷하지만 발효 방식이 다르다"
- 레시피에서 재료 클릭 시 glossary로 바로 연결
- 구매 안내: "주로 Asian grocery stores에서 구매 가능"
- 향후 커뮤니티가 확장하면 '글로벌 식재료 위키' 형태로 발전
- **고등급 인증 유저에게 편집 권한 부여** (위키 스타일 운영)

**Glossary 편집 정책 (단계적):**
- **MVP~V1:** Master 등급 유저가 직접 수정 가능 (단순 방식). 초기에는 Master 수가 적어 충돌 가능성 낮음.
- **V2:** Edit Proposal 방식으로 전환 — Master가 수정안을 제출하면 운영자 또는 다른 Master가 승인. Diff 기반으로 변경점 확인 가능. Rollback 기능 포함. 악의적 수정 방지.

### 3.4 현지 대체재 매핑 (Substitutes)

- **Visual Picker:** 특정 국가에서 구하기 힘든 재료(예: 깻잎)를 현지 대안(예: 시소)으로 시각적 제안
- **성공률 데이터:** 사용자들이 리믹스 과정에서 남긴 대체재 활용 후기를 데이터화하여 신뢰도 측정
- **주의:** 플랫폼이 정답을 주는 게 아님 → 사용자 경험 축적 구조
- **(향후) Local Grocery Finder:** 사용자 국가 정보 기반으로 해당 재료를 살 수 있는 현지 마켓(H-Mart 등)이나 온라인몰 링크 연결

### 3.5 Dietary Filter

- Vegan / Halal / Gluten-free / Nut-free 등
- 한식은 젓갈(fish sauce), 간장(soy=gluten) 등이 기본이라 차별화 포인트

**필터 UX 전략 (소프트/하드 분리):**
- **기본값:** 라벨 표시(Soft Filter) — 모든 레시피 보이되, 적합 여부를 마크로 표시. 부적합 레시피는 흐리게(Dimmed) 처리하거나 경고 문구 표시. 부적합 레시피도 "이 레시피의 [비건/할랄] 리믹스 버전이 있어요!" 연결 가능 → 리믹스 문화 활성화
- **설정 옵션:** 엄격한 필터링(Hard Filter) — 부적합 레시피 완전 숨김. 사용자가 직접 토글

### 3.6 레시피 재료 입력 UX: 연쇄 자동완성 (3-Step Chain)

레시피 등록 시 재료 입력은 **연쇄 자동완성** 방식. 재료 선택 → 수량 입력 → 단위 추천이 하나의 흐름으로 끊김 없이 이어짐. IDE 자동완성이나 리눅스 탭 완성과 유사한 경험.

**단계적 구현:**
- **MVP-α:** 기본 자동완성 드롭다운 → 수량 입력 → 단위 선택 (3단 폼). 핵심은 Ingredient ID 매칭이 처음부터 되는 것. 나중에 데이터 마이그레이션 지옥 방지.
- **V1:** 풀 스펙 3-Step Chain (키보드 탭 연쇄, 스마트 단위 추천, 모바일 최적화, 비정형 수량 버튼 등)

**입력 플로우:**

```
① 재료명 입력 (자동완성)
   [고추________]
    ├ 고추장      ← 2~3글자만 쳐도 후보 표시
    ├ 고춧가루    ← 탭/클릭으로 선택
    └ 청양고추

② 수량 입력 (선택 즉시 커서 이동)
   [고춧가루] [2_______]  ← 숫자 키보드 자동 전환 (모바일)

③ 단위 추천 (재료별 자주 쓰는 단위 우선)
   [고춧가루] [2] [______]
                   ├ 큰술    ← 이 재료의 common_units 기반 정렬
                   ├ 작은술
                   ├ g
                   └ 컵

④ 완료 → 칩으로 추가, 커서가 다음 재료 입력으로 자동 이동
   🏷 고춧가루 · 2큰술  ✕
   [다음 재료 입력______]
```

**전체 과정이 키보드에서 손 안 떼고 끝남:**
```
"고추" → 탭(선택) → "2" → 탭(선택) → 완료 → 다음 재료...
```

**모바일 UX:**
- 재료명: 검색 자동완성 드롭다운 터치 선택
- 수량: 숫자 키패드 자동 팝업 (`inputmode="decimal"`)
- 단위: 드롭다운 대신 **버튼 나열**로 터치 최적화
- 한 재료당 3~4초에 완료

**비정형 수량 처리 ("약간", "적당히"):**
- 수량 입력 단계에서 숫자 외에 비정형 표현도 버튼으로 제공: `약간 | 적당히 | 한줌 | to taste`
- 비정형 선택 시 단위 단계 자동 스킵
- 내부 저장: `{ "amount": null, "unit": null, "qualifier": "to_taste" }`

**분수 입력 ("1/2", "반"):**
- `1/2` 입력 허용 → 내부적으로 `0.5`로 normalize
- 표시는 원문 유지 옵션 가능 ("1/2" 그대로 보여주기)

**다국어 대응:**
- 사용자는 현재 UI 언어로 검색/선택 (한국어 UI → "고추장", 영어 UI → "gochujang")
- 단위도 DB 기반 다국어 라벨로 표시 (한국어: "큰술", 영어: "tbsp", 일본어: "大さじ")
- 언어 추가 시 DB에 번역만 추가, 정규식/파싱 규칙 추가 불필요

**하나의 Ingredient ID가 연결하는 것들:**

```
Universal Ingredient ID (ing_gochujang_001)
├→ Glossary: 재료 클릭 → 설명 페이지
├→ 단위 변환: 2 tbsp → 30g 자동 계산
├→ 구매 링크: 미국 → Amazon / 한국 → 쿠팡
├→ Dietary Filter: 글루텐 포함 → 글루텐프리 필터에서 제외
├→ 쇼핑 리스트: 떡볶이 고추장 2T + 비빔밥 고추장 1T = 3T 합산
├→ 냉장고 털이: "고추장 있어요" → 고추장 쓰는 모든 레시피 매칭
└→ 연쇄 추천: 떡볶이 후 남은 고추장 → "고추장 불고기도 만들어보세요"
```

**DB에 없는 재료 처리 (엣지 케이스):**
- 기본: DB 자동완성으로 선택 유도
- Fallback: 못 찾으면 자유 텍스트 입력 허용 + "새 재료 등록 요청" 버튼
- 자유 입력 재료는 Glossary/구매 링크/연쇄 추천 등 기능 제한됨
- 운영자 승인 후 DB에 정식 등록 → 기존 레시피에 자동 매칭 시도
- Master 등급 유저: 직접 DB에 새 재료 등록 가능 (Glossary 편집권)

### 3.7 Ingredient 데이터 모델

```json
{
  "id": "ing_gochugaru_001",
  "names": { "ko": "고춧가루", "en": "Gochugaru", "ja": "唐辛子粉" },
  "category": "양념",
  "common_units": ["tbsp", "tsp", "g", "cup"],
  "default_unit": "tbsp",
  "is_commodity": false,
  "dietary_flags": ["vegan", "gluten_free"],
  "substitutes": ["ing_paprika_001"],
  "affiliate_links": { "us": "...", "kr": "...", "global": "..." }
}
```

**Unit 데이터 모델 (다국어 단위):**

```json
{
  "unit_id": "tbsp",
  "names": {
    "ko": ["큰술", "큰 술", "T"],
    "en": ["tbsp", "tablespoon", "Tbsp"],
    "ja": ["大さじ"]
  },
  "ml_equivalent": 15
}
```

단위를 DB화하면 다국어 변환이 자동. 새 언어 추가 = DB에 번역만 넣으면 끝.

**레시피 내 재료 참조 구조:**

```json
{
  "ingredient_id": "ing_gochugaru_001",
  "amount": 2,
  "unit": "tbsp",
  "qualifier": null,
  "note": "매운맛 기준"
}
```

| 필드 | 용도 |
|------|------|
| `ingredient_id` | Universal Ingredient ID 참조 |
| `amount` | 수량 (null = 비정형) |
| `unit` | 단위 unit_id (null = 비정형) |
| `qualifier` | 비정형 수량: "to_taste", "pinch", "handful" 등 |
| `note` | 작성자 메모 (선택) |

### 3.8 초기 재료 DB 시드

**MVP-α에서 핵심 한식 재료 30개를 풀 데이터로 등록.** 수보다 질이 중요 — 재료당 다국어 이름, Glossary 설명, common_units, dietary_flags, 대체재, 구매 링크까지 완벽하게 채운 30개가 어설픈 100개보다 나음. 나머지는 자유 텍스트 fallback + 점진적 DB 등록.

```
장류/양념 (12): 고추장, 된장, 간장, 참기름, 고춧가루, 들기름, 식초, 맛술,
               물엿, 참깨, 소금, 설탕
곡물/면 (4):    쌀, 떡볶이떡, 당면, 소면
채소 (7):       배추, 무, 대파, 마늘, 양파, 감자, 당근
단백질 (5):     두부, 소고기, 돼지고기, 닭고기, 계란
기타 (2):       배추김치, 어묵
```

시드 30개 선정 기준: 시드 레시피 10~15개에 등장하는 재료를 역산하여 선정.
MVP-β 이후 유저 요청 + 레시피 추가에 따라 점진적 확장 → V1 시점 50~80개 목표.

## 4. 핵심 시스템 2: Fork & Remix 생태계 (The Evolution of Taste)

### 4.1 기본 구조

- 쿡패드 츠쿠레포 + GitHub fork 개념
- 원본 한식 레시피에서 자기만의 퓨전 버전을 "remix"로 올림
- **리믹스 자체는 항상 자유** — 누구나 아무 레시피를 리믹스 가능

### 4.2 계층 구조 및 승인권 (Hierarchy & Approval)

**레시피 트리(Recipe Tree):** 오리지널(Root) → 리믹스(Branch) → 리믹스의 리믹스

```
🌳 오리지널 떡볶이 (Root)
├── 라볶이 (1차 리믹스) ← 오리지널 제작자가 승인
│   ├── 치즈 라볶이
│   └── 마라 치즈 라볶이 ← 라볶이 제작자가 승인
├── 짜장 떡볶이
└── 로제 떡볶이 ⭐ Master's Choice
```

**직계 부모 승인권:**
- "마라 치즈 라볶이"의 공식 리믹스 여부는 **직계 부모인 '라볶이' 제작자**가 결정
- 오리지널(Root) 제작자가 아닌 직계 부모가 결정하는 이유:
  - 확장성: 한 사람에게 병목 방지
  - 전문성: 중간 단계 제작자가 해당 영역을 더 잘 앎

**뿌리 제작자(Root)의 명예:**
- 가문 전체에서 뛰어난 레시피에 **'Master's Choice'** 배지 부여 권한
- 트리 전체의 파생 현황 통계 확인 가능

**핵심 원칙:** 승인권은 "공식 리믹스로 추천할지"에 한정. 리믹스 생성 자체를 막는 것은 아님.

**트리 깊이 제한 정책:**
- Soft cap: **5~7단계** — 이 이상은 "계보 더보기"로 접어서 표시
- 깊이 제한은 리믹스 생성을 막는 것이 아님 (생성은 무제한), UI 표시 방식만 제어
- 목적: UX 가독성 유지 + 트리 조회 성능 보호
- 구현은 V2 (트리 시각화 도입 시점)에서, 실제 데이터 깊이를 보고 최종 수치 결정

### 4.3 저작권 보호 및 예우

**Inspired by 시스템:**
- 타인의 레시피를 참고했을 때 출처를 자동 명시
- 레시피 입력 폼에 'Inspired by' / 'Reference' 필드 제공
- 플랫폼 내 리믹스: "이 레시피는 [원본]의 remix입니다" 자동 링크
- 외부 참고: "Inspired by: [Maangchi's YouTube Channel]" 수동 입력

**Safe Harbor / DMCA 대응:**
- 미국 DMCA + 한국 저작권법(OSP 의무) 이중 준수
- 신고 접수 시 즉시 삭제(Notice and Takedown) 프로세스
- 사용자 신고 시스템을 MVP 단계부터 포함

**이용약관(ToS) 핵심 조항:**
- 레시피 올릴 때 "다른 사용자가 이 레시피를 Remix하는 것을 허용한다" 동의 (Creative Commons 스타일)
- 콘텐츠 저작권은 사용자에게 있되, 플랫폼이 서비스 홍보/다국어 번역에 사용할 권한 보유
- 준거법: 대한민국 법 (약관에 명시)

**레시피 저작권 참고:**
- 재료 목록과 분량(수치)은 저작권 보호 대상이 아님 (사실/데이터 영역)
- "고추장 1스푼"을 "고추장: 1스푼"으로 구조화하는 것은 데이터 변환이므로 문제 없음
- 기능적 조리 순서("물이 끓으면 면을 넣으세요")도 창작성 인정 어려움
- **보호 대상:** 사진, 영상, 독창적 설명 텍스트(미사여구/에피소드/팁), 레시피 모음집
- 유튜브/방송 레시피의 재료+분량을 구조화 등록하는 것은 법적으로 문제 없음
- 단, 설명 텍스트는 본인의 말로 재구성 필수, 사진은 직접 촬영 필수
- **출처 표기:** 법적 의무는 아니지만 "○○의 레시피를 참고하여 만들었습니다" 식으로 밝히면 신뢰도 향상 + 마찰 방지

**국제 저작권 원칙:**
- 베른 협약(Berne Convention)에 따라 '보호가 요청되는 국가의 법' 적용 (보호국법 원칙)
- 서버가 한국에 있어도 서비스가 닿는 모든 국가의 저작권자로부터 소송 가능성 있음
- 대응: DMCA 수준의 '신고 및 즉시 삭제' 절차를 갖추는 것이 가장 안전

### 4.4 구현 로드맵

- **V1:** "이 레시피는 [원본]의 remix입니다" 링크로 심플하게
- **V2:** 트리 시각화, 승인권 시스템, Master's Choice 배지

### 5.1 축별 평가

- **Difficulty** (난이도)
- **Authenticity** (정통성)
- **Kid-friendly** (아이 적합도)

### 5.2 "I made this!" 후기

- 사진 필수 후기
- 리뷰 기반 레시피 신뢰도 스코어

### 5.3 사진 신뢰도 시스템 (4단계)

| 단계 | 검증 방법 | 목적 |
|------|----------|------|
| 1단계 (자동) | **pHash 검사** | 플랫폼 내 기존 사진과의 중복 여부 즉시 확인 |
| 2단계 (자동) | **EXIF 분석** | EXIF 존재 여부(has_exif) + 기기 유형(device_category)만 저장 → **신뢰 점수 반영** (차단 X) |
| 3단계 (커뮤니티) | **"I made this!" 리뷰** | 다른 유저들이 실제 결과물 보고 남기는 리뷰 기반 검증 |
| 4단계 (사후) | **유저 신고** | 저작권 침해 의심 시 운영자에게 신고 |

**EXIF 주의사항:** 블로거가 자기 블로그에서 사진을 다운받아 올리면 EXIF가 삭제된 상태일 수 있음. 따라서 EXIF 없음 = 차단이 아니라 "일반 유저" 처리. EXIF 있음 = "직접 촬영 확률 높음" 인증 마크 부여.

**워터마크 자동 삽입 (선택적):** 사용자가 사진 업로드 시 '맛담' 로고 + 사용자 ID를 워터마크로 자동 삽입하는 옵션 제공 → 외부 무단 유출 심리적 방지

## 6. 사용자 기능 및 UX

### 6.0 레시피 작성 전체 플로우

```
1️⃣ 제목 입력
   [떡볶이 _________________]

2️⃣ 재료 입력 (연쇄 자동완성)
   재료 선택 → 수량 → 단위 → 칩 추가 → 다음 재료
   (섹션 3.6 참고)

3️⃣ 조리법 입력 (슬라이드 방식)
   카드 한 장 = 한 스텝, Enter 2번 = 다음 스텝
   각 카드: 텍스트 + 타이머(선택) + 사진(선택) + 재료 태그(선택)
   (섹션 6.2 참고)

4️⃣ 미리보기 (= Cooking Mode 프리뷰)
   작성한 그대로 보임 — 작성 화면과 소비 화면이 동일

5️⃣ 저장
```

**레이아웃:**
- 모바일: 세로 스크롤 (제목 → 재료 → 조리법)
- 데스크톱: 좌측 재료 + 우측 슬라이드 병렬

**재료와 조리법이 분리되어야 하는 이유:** 실제 요리 과정이 "재료 목록 보고 장보기 → 재료 준비 → 조리법 따라 요리" 순서이며, 쇼핑 리스트/단위 변환/냉장고 털이/연쇄 추천 등 모든 기능이 독립된 재료 목록 위에서 작동.

### 6.1 쿠킹 모드 (Cooking Mode) ⭐

플랫폼 차별화의 핵심 기능. 만개의레시피, 해먹남녀에도 없음.

**기본 구조:**
- 일반 모드: 전체 레시피가 한 페이지에 (읽기/미리보기용)
- 쿠킹 모드 진입: 풀스크린 전환, 스텝별 카드 한 장씩 (포스트잇/카드 느낌)

**카드별 구성:**
- 그 스텝에서 필요한 재료만 하이라이트
- 해야 할 동작 한두 문장
- 타이머 필요 시 바로 시작 버튼
- 좌우 스와이프 또는 큰 "다음" 버튼

**디바이스별 반응형 레이아웃:**

| 화면 크기 | 레이아웃 | 비고 |
|-----------|----------|------|
| ~480px (폰) | 현재 스텝 풀카드 + 하단에 다음 스텝 첫 줄 peek | 위로 스와이프하면 전환 |
| 481~1024px (태블릿) | PPT 발표자 보기 스타일 — 상위 70~75% 현재 스텝, 하단 25~30% 다음 스텝 미리보기 | 다음 스텝은 흐리게 처리 |
| 1025px~ (데스크톱) | 좌우 분할 가능 | |

**기기 판별:** `window.innerWidth` 기반 반응형 (UA 파싱 비추). 기기를 맞추는 게 아니라 "화면이 충분히 넓은가?"를 보는 것.

**부가 기능:**
- Wake Lock API (`navigator.wakeLock`) — 화면 안 꺼지게
- 복수 타이머 관리 — 카드 넘겨도 이전 스텝 타이머 유지, 플로팅 타이머 표시
- 손 젖었을 때 대응 — 큰 버튼, 화면 아무 곳 탭으로 넘기기
- (향후) Web Speech API 음성 "다음"

**PWA 지원 (단계적):**
- **V1:** PWA manifest + 홈 화면 추가 — 앱처럼 실행 가능. 구현 간단.
- **V2:** Service Worker 기반 오프라인 캐싱 — 최근 본 레시피 로컬 저장, 주방에서 네트워크 끊겨도 쿠킹 모드 사용 가능. 이미지 캐싱, 스토리지 관리, 동기화 로직 등 기술적 복잡도가 높아 V2로 미룸.
- 오프라인 모드에서는 읽기 전용 (리뷰 작성, 구매 링크 등은 온라인 필요)
- 이거 하나만 해도 "진짜 요리용 플랫폼"으로 차별화

### 6.2 레시피 조리법 입력: 슬라이드 방식 ⭐

**핵심 원칙: 작성 화면 = Cooking Mode 프리뷰.** 작성할 때 보는 카드가 곧 요리할 때 보는 카드. 사용자가 자연스럽게 "여기서 한 동작 끝" 단위로 사고하게 유도.

**입력 규칙:**
- Enter 1번 → 같은 스텝 내 줄바꿈
- Enter 2번 (빈 줄) → 다음 스텝으로 전환
- 모바일: [다음 스텝 →] 버튼도 제공

**모바일 작성 화면:**

```
┌─────────────────────────────────────┐
│  Step 1 / ?                          │
│  ┌───────────────────────────────┐  │
│  │                               │  │
│  │  두부를 1cm 두께로 썬다        │  │
│  │  키친타월로 물기를 제거한다     │  │
│  │                               │  │
│  └───────────────────────────────┘  │
│  ⏱ 타이머: [없음 ▼]   📷 [사진 추가] │
│  🥘 재료: [두부] [식용유]            │  ← 재료 태그 (선택)
│                                     │
│  ← prev          ● ○ ○    [다음 →]  │
│                                     │
│  💡 Enter 2번 → 다음 스텝            │
└─────────────────────────────────────┘
```

**데스크톱 작성 화면 (좌측 목록 + 우측 편집):**

```
┌──────────────────────────────────────────────────┐
│ 📝 제목: [떡볶이                                ] │
├──────────────────┬───────────────────────────────┤
│ 📦 재료           │  🍳 조리법                     │
│                  │                               │
│ [재료 추가 🔍  ] │  ┌───────────────────────┐    │
│                  │  │ Step 2                 │    │
│ 🏷 고추장 2큰술  │  │                        │    │
│ 🏷 떡 300g      │  │  팬에 기름을 두르고      │    │
│ 🏷 어묵 3장     │  │  중불에서 앞뒤 3분씩     │    │
│ 🏷 대파 1뿌리   │  │  굽는다                 │    │
│                  │  │                        │    │
│                  │  │  ⏱ 3분  📷  🥘 두부    │    │
│                  │  └───────────────────────┘    │
│                  │    ● ● ○ ○                    │
│                  │                               │
│                  │  💡 좌측 재료를 스텝에          │
│                  │     드래그하면 연결됩니다       │
└──────────────────┴───────────────────────────────┘
```

**각 스텝 카드 요소:**
- 텍스트 (자유 작성)
- 타이머 (선택, 시간 패턴 "3분", "5 min" 감지 시 자동 제안)
- 사진 (선택, 스텝별 1장)
- 재료 태그 (선택, 해당 스텝에서 사용하는 재료 연결)

**붙여넣기 호환:**
- 기존 레시피 텍스트를 통째로 붙여넣으면 빈 줄(줄바꿈 2개) 기준으로 자동 스텝 분리
- 줄바꿈 1개 = 같은 스텝 내 줄바꿈으로 처리

**스텝 관리:**
- 하단 dot indicator (● ● ○ ○)로 전체 스텝 수 파악
- 좌우 스와이프 또는 화살표로 스텝 간 이동
- 드래그로 스텝 순서 변경
- 빈 스텝은 저장 시 자동 삭제
- 8~10줄 넘어가면 "다음 스텝으로 나누시겠어요?" 제안

**Cooking Mode와의 연결:**
- 구조화 완료된 레시피 → **Full Cooking Mode** (카드 1장 = 1 step, 타이머, 재료 하이라이트)
- 구조화 안 된 레시피 (외부 유입 등) → **Lite Cooking Mode** (문단 단위 카드, 타이머 없음, 간단 next 버튼만)

**데이터 구조:**

### 6.3 지능형 추천 시스템

**식재료 연쇄 추천 (Ingredient Chaining) — 선제적 추천:**
- 떡볶이를 만들고 남은 양배추 3/4통을 버리지 않도록, 이어서 만들 수 있는 최적의 레시피를 선제적으로 제안
- 레시피 하단 또는 쿠킹 모드 완료 화면에 추천 카드 배치
- Universal Ingredient ID 기반 매칭
- 사용자의 Dietary Filter + 난이도 고려
- **현지 구매 단위 반영:** 한국은 양배추 1/4통 판매 가능하지만, 미국/유럽은 통째로 판매 → "통째로 샀을 때" 기준으로 남은 양 계산하면 외국인 UX 향상

**냉장고 털이 — 사후 검색형:**
- 보유 재료를 다국어 ID 기반으로 검색
- 사용자의 식단(비건, 할랄 등)에 맞는 최적의 리믹스 버전 노출

**연쇄 추천 vs 냉장고 털이:**
- 연쇄 추천 = Proactive: "이 요리 하면 이만큼 남을 테니, 다음엔 이걸 하세요"
- 냉장고 털이 = Reactive: "이미 남은 재료가 있는데 뭘 하지?"

### 6.4 Shopping List 생성

- 여러 레시피 골라서 "이번 주 장보기 리스트" 자동 생성
- 재료 겹치면 합산 (Universal Ingredient ID 활용)

### 6.5 Recipe Linter (품질 검수) — 경쟁 플랫폼에 거의 없는 차별화 기능

레시피 등록 전 자동 품질 체크:

**기본 검사:**
- 필수 단위(Metric/Imperial) 누락 여부 (예: "180도 오븐" → "180°C (356°F)" 권장)
- 모호한 설명 감지 (예: '적당히', '한 꼬집', '약간' → 구체적 수치 권장)

**재료-스텝 정합성 검사 (Universal Ingredient ID 기반):**
- 스텝에 등장하는 재료가 재료 목록에 없으면 경고 ("Step 3에서 '참기름'을 사용하지만 재료 목록에 없습니다")
- 재료 목록에는 있지만 어떤 스텝에서도 사용되지 않는 재료 경고

**스텝 품질 검사:**
- 스텝당 동사 최소 1개 포함 검사 (빈 스텝이나 메모만 있는 스텝 방지)
- 타이머가 필요할 것 같은 키워드 감지 ("끓이다", "볶다", "오븐" 등) → 타이머 설정 권장

**경고 수준:** 모두 경고(warning)이며 차단(block)하지 않음. 작성자가 무시하고 등록 가능.

### 6.6 온보딩 플로우 (Dietary Preferences 수집)

| 단계 | 방식 | 내용 |
|------|------|------|
| Step 1 | 간편 가입 | 소셜 로그인(Google, Apple)으로 최소 정보만 |
| Step 2 | 선택적 온보딩 | "맞춤 한식을 찾아드릴까요?" 팝업으로 비건/할랄/난이도 체크 유도 |
| Step 3 | 프로필 설정 | 언제든 마이페이지에서 선호도 수정 가능 |

수집 항목: 식단 제한(Vegan, Halal, Gluten-free, Nut-free), 요리 숙련도(Beginner~Master), 선호 언어

### 6.7 게이미피케이션 (Gamification)

**글로벌 활동 표시 (단계적 도입):**
- **초기 (MAU 1,000 미만):** 전 세계 지도 대신 **"최근 활동 국가 3곳" 카드 UI**로 시작. 텅 빈 지도는 심리적으로 치명적이므로 절대 먼저 보여주지 않음.
  - "Now Cooking In..." 실시간 알림: "A user in France just remixed Kimchi Stew!" 토스트/띠 형태
  - Country Leaderboard: "Most Active Countries This Week" 상위 3~5개국 국기+순위 카드
- **MAU 1,000+ 이후:** 인터랙티브 히트맵 도입. 레시피 많은 지역 진한 색 → 클릭 시 해당 국가 레시피 검색.
- **데이터 집계:** 매번 실시간 count 대신 일정 주기로 캐싱하여 메인에 표시 (DB 부하 방지)

**사용자 등급 및 인센티브:**

| 등급 | 조건 | 주요 권한 |
|------|------|----------|
| Beginner | 가입 | 레시피 등록, 리뷰 작성 |
| Intermediate | 인증된 사진 N개 + 리뷰 N개 | 검색 결과 우선 노출, 프로필 커스텀 |
| Master | 꾸준한 인증 콘텐츠 + 리믹스 활동 | **Glossary 편집권**, 리믹스 승인권, 분석 대시보드 |

**인센티브 체계:**
- 인증 사진 포함 레시피 → 검색 결과/메인 페이지 우선 노출
- "I made this!" 인증 리뷰 → 리뷰 영역 상단 고정
- 리믹스 통계 제공 — 자신의 레시피가 전 세계 몇 개국으로 뻗어나갔는지
- (향후) 포트폴리오 페이지 — 프로필을 개인 포트폴리오로 활용, 전용 URL 제공
- (향후) 공식 SNS 피처링 — 고퀄리티 인증 레시피를 TikTok/Instagram에서 홍보

**Kitchen Crew (그룹 리믹스):**
- 특정 테마(예: "뉴욕에서 즐기는 추석 음식")를 위해 여러 유저가 팀을 이뤄 하나의 Remix Tree를 완성하는 프로젝트형 메뉴

## 9. 기술 설계 및 인프라

### 9.1 기술 스택

| 영역 | 기술 | 이유 |
|------|------|------|
| **Frontend** | Next.js (App Router) | 글로벌 SEO, 다국어 라우팅(next-intl), SSR/SSG |
| **Backend/DB** | Supabase (PostgreSQL) | 관계형 트리 구조(Recursive Query), jsonb로 유연한 Preferences, Auth/Storage 통합 |
| **Styling** | Tailwind CSS | 쿠킹 모드 등 기기별 복잡한 반응형 UI 빠른 구현 |
| **Search** | Typesense 또는 Meilisearch | 다국어 오타 교정 (Kimbap→Gimbap), 빠른 검색 |
| **Infrastructure** | Vercel | 글로벌 엣지 네트워크, Edge Functions, Image Optimization |
| **인증** | Supabase Auth | 소셜 로그인 (Google, Apple) |

**Supabase 유지 근거:**
- 단순히 무료라서가 아니라, PostgreSQL의 Recursive Query가 레시피 트리 추적에 강력
- Auth + Storage + Realtime 통합 환경으로 1인 개발 시 DevOps 부담 최소화
- 무료 티어로 MVP, 트래픽 증가 시 Pro로 전환
- 글로벌 엣지 네트워크를 통한 빠른 응답 (개인 NAS 대비 압도적 속도 차이)

**대체제 참고:** PlanetScale(MySQL) — 전 세계 수백만 동시 접속 수준의 수평 확장이 필요한 경우 고려. 단, 초기 개발 속도와 통합 환경은 Supabase가 유리.

**다국어/현지화 구현:**
- Next.js Internationalized Routing: `/en/recipe/bulgogi`, `/ko/recipe/bulgogi`
- 정적 텍스트(메뉴, 버튼): next-intl 처리
- 동적 데이터(레시피 본문, 재료명) 다국어 전략:
  - **MVP~V1: jsonb 방식** — `{ "ko": "고추장", "en": "Gochujang" }` 형태로 한 컬럼에 저장. 빠르고 간단. 재료 50~100개, 지원 언어 2~3개 수준에서 충분.
  - **V2: translation 테이블 분리** — 커뮤니티 번역 도입 시 `ingredient_translations (ingredient_id, locale, name, description)` 테이블로 마이그레이션. 언어별 기여자 추적, 번역 품질 관리 가능.
  - 문서 전체에서 다국어 데이터는 이 기준을 따름.

### 9.2 쿠킹 모드 기술 포인트

- 디바이스 분기: CSS media query / `window.innerWidth`
- 화면 유지: Wake Lock API
- 타이머: 복수 타이머 관리, 카드 전환 시에도 유지
- 터치 최적화: 큰 터치 타겟, 스와이프 제스처
- 터치 감지: `'ontouchstart' in window` (보조 수단)

### 9.3 레시피 DB 스키마 (초안)

```
Recipe
├── recipe_id (PK)
├── title (다국어 jsonb)
├── author_id (FK → User)
├── parent_recipe_id (FK → Recipe, nullable) ← 직계 부모 Remix 관계
├── parent_version_number (INT, nullable) ← 리믹스 시점의 부모 버전
├── root_recipe_id (FK → Recipe, nullable) ← 트리 최상위
├── version_number (INT, default 1) ← 수정 시 증가
├── ingredients[] ← Universal Ingredient ID 참조
│   ├── ingredient_id
│   ├── amount (nullable, 비정형 시 null)
│   ├── unit (unit_id 참조, nullable)
│   ├── qualifier (nullable: "to_taste", "pinch", "handful" 등)
│   └── note (nullable: 작성자 메모 "매운맛 기준" 등)
├── steps[]
│   ├── order
│   ├── description
│   ├── ingredients_used[]
│   ├── timer_seconds
│   └── image_url
├── dietary_tags[] (vegan, halal, gluten-free...)
├── difficulty_level (beginner/intermediate/master)
├── inspired_by (text, nullable) ← 외부 참고 출처
├── is_approved_remix (boolean) ← 직계 부모 승인
├── master_choice (boolean) ← Root 제작자 배지
└── created_at / updated_at

Recipe_Versions (수정 이력 보존)
├── version_id (PK)
├── recipe_id (FK → Recipe)
├── version_number (INT)
├── title (jsonb)
├── ingredients (jsonb) ← 해당 버전의 스냅샷
├── steps (jsonb) ← 해당 버전의 스냅샷
└── created_at

User Profile (Supabase auth.users 연동)
├── user_id (PK)
├── display_name
├── country
├── tier (beginner/intermediate/master)
├── trust_score (float)
├── preferences (jsonb) ← { dietary: [...], skill_level: "...", preferred_unit: "..." }
└── created_at
```

**버전 관리 정책:**
- 레시피 수정 시 `version_number` 증가 + 이전 버전을 `recipe_versions`에 스냅샷 저장
- 리믹스는 `parent_version_number`로 "어떤 버전을 기반으로 만들었는지" 고정
- 원본이 수정되어도 리믹스의 참조 버전은 변하지 않음
- MVP: `version_number` 필드만 운영, 스냅샷 테이블은 V1에서 도입

## 10. 마케팅 및 운영 전략

### 10.1 콜드 스타트 돌파 — 가장 큰 리스크이자 최우선 과제

초기 포지셔닝은 "커뮤니티 플랫폼"보다 **"큐레이션 사이트"** 느낌으로 시작. 시드 콘텐츠가 20개 이하면 플랫폼이 아니라 개인 블로그처럼 보이므로, "엄선된 한식 레시피 + 리믹스 본보기"로 신뢰감을 먼저 쌓는다.

### 10.2 시드 콘텐츠 전략

- MVP-α: 10개, MVP-β: +10개 = **총 20개** 고퀄리티 오리지널 한식 레시피
- **영어 품질 최우선** — 어색한 영어는 외국인 타겟 전략을 무너뜨림
- 시드 콘텐츠는 수보다 질. **반드시 직접 요리한 사진 포함** (AI 대응 차별점)
- **"이 플랫폼은 이렇게 노는 곳이다" 본보기:**
  - 원본: 정통 된장찌개
  - 리믹스 1: (미국 현지) 주키니와 파마산 치즈를 넣은 된장찌개
  - 리믹스 2: (비건) 멸치 육수 대신 채수를 사용한 된장찌개

### 10.3 Phase 0: 런칭 전 씨뿌리기 (개발과 병행, 2~4주)

플랫폼 개발 중에 **동시에** 진행:

- r/KoreanFood, r/cooking, r/asiancooking에 **개인 계정으로 활동 시작**

- 이 과정에서 "한식 리믹스"에 대한 실제 관심도와 니즈를 직접 관찰
- (선택) Discord 서버 먼저 열어서 관심 있는 사람 10~20명 모으기

### 10.4 Phase 1: 소프트 런칭 (MVP-α 완성 시점)

- Reddit에 "한식 레시피를 리믹스하고 공유하는 사이트를 만들었다" 글 하나 올리기
- 반드시 **가치를 먼저 제공:** "이 사이트에서 된장찌개를 5가지 버전으로 비교할 수 있다"

### 10.5 Phase 2: 콘텐츠 엔진 (1~3개월)

- TikTok/Reels 30~60초 영상 → "Full recipe on [사이트]" **(2명 이상일 때만 현실적)**

### 10.6 Phase 3: 커뮤니티 부스팅 (3~6개월)

### 11.2 제휴 마케팅 상세 (핵심 수익원)

**레시피 상세 페이지 구매 섹션:**

레시피의 재료 목록에서 `is_commodity === false`인 특수 재료만 자동으로 구매 링크 표시. 범용 재료(설탕, 물, 소금 등)는 링크 안 달음.

```
🛒 이 레시피에 필요한 재료
┌─────────────────────────────────────┐
│ ✅ 고추장 (CJ 해찬들 500g)    → Buy │  ← affiliate link
│ ✅ 떡볶이 떡 (풀무원 1kg)     → Buy │  ← affiliate link
│ ✅ 어묵 (삼진 사각어묵)       → Buy │  ← affiliate link
│ ☐ 설탕                              │  ← is_commodity, 링크 없음
│ ☐ 물                                │
│                                     │
│        [Buy All on Amazon] 🔗       │  ← 한번에 장바구니
└─────────────────────────────────────┘
```

Universal Ingredient ID에 affiliate_links 필드 추가:

```json
{
  "id": "ing_gochujang_001",
  "is_commodity": false,
  "affiliate_links": {
    "us": { "url": "https://amazon.com/dp/xxx?tag=matdam-20", "product_name": "CJ Haechandle Gochujang 500g" },
    "uk": { "url": "https://amazon.co.uk/dp/xxx?tag=matdam-21", "product_name": "..." },
    "kr": { "url": "https://link.coupang.com/xxx", "product_name": "CJ 해찬들 고추장 500g" },
    "global": { "url": "https://iherb.com/xxx?rcode=MATDAM", "product_name": "..." }
  }
}
```

재료 하나당 한번만 링크 설정 → 그 재료가 들어간 모든 레시피에 자동 적용.

**별도 Shop 페이지 (V2):**
- GNB에 "Shop" 메뉴 추가
- 한식 재료 전체를 카테고리별로 큐레이션
- 레시피에서 바로 사는 유저 + 식재료 쇼핑 목적 유저 동시 포착

**국가별 자동 리다이렉트:**
- 유저 접속 IP 기반으로 해당 국가 제휴 링크 자동 삽입
- 미국 IP → Amazon.com / 영국 IP → Amazon.co.uk / 한국 IP → 쿠팡 / 기타 → iHerb (fallback)

**SEO 리스크 관리:**
- 구매 섹션은 **accordion(접힘) 기본 닫힘** — 레시피 콘텐츠가 본체, 구매는 부가 기능임을 명확히
- 모든 affiliate 링크에 `rel="nofollow sponsored"` 적용 — Google 패널티 방지
- Glossary의 "Where to buy"는 정보 중심으로 작성 (단순 링크 나열 X)

### 11.3 글로벌 제휴 프로그램 & 정산

각 나라 계좌 불필요. 대형 어필리에이트 프로그램은 글로벌 정산 지원.

| 프로그램 | 가입 | 한국 계좌 수령 | 비고 |
|---------|------|--------------|------|
| Amazon Associates (US) | 미국 프로그램 별도 가입 | ✅ 국제 송금 가능 | OneLink로 글로벌 자동 리다이렉트 |
| Amazon Associates (UK/JP 등) | 각국 별도 가입 | ✅ 각각 국제 송금 | |
| iHerb Affiliate | 글로벌 단일 가입 | ✅ PayPal/은행 송금 | 아시안 식재료 다수 |
| CJ Affiliate / Impact | 글로벌 단일 가입 | ✅ PayPal/은행 송금 | H-Mart, Weee! 등 입점 |
| 쿠팡 파트너스 | 한국 가입 | ✅ 한국 계좌 | 한국 유저 전용 |

**PayPal 계정 하나 만들어두면** 대부분의 글로벌 어필리에이트 정산을 한 곳에서 수령 가능.

## 13. 우선순위 로드맵

### 13.1 Phase 요약

| Phase | 내용 | 기간 (2명 기준) |
|-------|------|----------------|
| **MVP-α** | 레시피 CRUD, 재료 자동완성(기본 드롭다운), 단위 변환, 기본 리믹스(링크), 소셜 로그인, DMCA 이메일 접수, 시드 콘텐츠 10개, SEO | 4~6주 |

| **V1** | 슬라이드 입력 고도화, 연쇄 자동완성 풀 스펙, 쿠킹 모드(+PWA), 구조화 리뷰, pHash, 신고 UI, 쇼핑 리스트, 연쇄 추천, 냉장고 털이, Recipe Linter, 온보딩 | +2~3개월 |
| **V2** | 레시피 트리 시각화, 공식 인증 시스템, 글로벌 활동 표시(MAU 1,000+), AI 파싱, PWA 오프라인 캐싱, Glossary edit proposal, 국가별 퀴진 확장 | 커뮤니티 성장 후 |

**MVP를 α/β로 나누는 이유:** 1인~2인 개발로 현재 MVP 범위를 2개월 안에 끝내기는 빡빡함. α에서 핵심만 만들어 Reddit에 뿌리고 반응을 본 뒤, β에서 피드백 반영하여 보강. MVP에서 검증할 건 딱 하나: **"외국인이 한식 리믹스 레시피를 올리고 싶어하는가?"**

### 13.2 MVP-α 구현 순서 — "사람이 써볼 수 있는 상태"

**Step 1: 인프라 + 인증 기반 (1~2주)**
- Next.js 프로젝트 세팅 (App Router, Tailwind, next-intl, **영어 기본 locale**)
- Supabase 연동 (DB, Auth, Storage)
- 소셜 로그인 (Google) + 기본 User Profile 테이블
- 배포 파이프라인 (Vercel)

**Step 2: 재료 엔진 기초 (1주)**
- Universal Ingredient ID 테이블 설계 + 시드 데이터 (**핵심 30개, 풀 데이터**)
- 단위 변환 유틸리티 함수 (Metric ↔ Imperial)
- 재료 자동완성 드롭다운 (기본 검색 + 선택 → 수량 → 단위 3단 폼)

→ *Glossary 페이지는 MVP-β로. 자동완성과 Ingredient ID 매칭이 먼저.*

**Step 3: 레시피 CRUD + 슬라이드 입력 기본 (2주)**
- 레시피 DB 스키마 (recipe, steps, ingredients_used, version_number)
- 레시피 등록: 슬라이드 방식 기본 버전 (Enter 2번 = 다음 스텝, 카드별 타이머/사진)
- 레시피 상세 페이지 (재료 목록, 스텝, 단위 토글)
- 레시피 목록/탐색 페이지 (검색, 난이도 필터)

→ *데이터 구조는 처음부터 슬라이드/구조화 기반으로 확정. 나중에 마이그레이션 방지.*

**Step 4: 리믹스 기본 (0.5주)**
- parent_recipe_id / root_recipe_id 필드
- "Remix this recipe" 버튼 → 원본 데이터 복사 후 수정
- "Inspired by [원본]" 자동 링크 표시

**Step 5: 시드 콘텐츠 + SEO + 런칭 (1주)**
- 10개 고퀄리티 오리지널 한식 레시피 직접 제작 (영어 우선)
- 리믹스 본보기 1~2개 (정통 → 현지화)
- SEO 메타데이터 (SSR, og:image, 구조화 데이터)
- Privacy Policy + DMCA 이메일 접수 (푸터)
- 랜딩 페이지 마무리
- **Reddit 소프트 런칭**

### 13.3 MVP-β 구현 순서 — "반응 보고 붙이기"

α에서 유저 피드백을 받은 후 우선순위 조정.

**Step 6: 리뷰 + 북마크**
- 단순 별점 (5점) + 텍스트 리뷰
- "I made this!" 사진 업로드 (선택)
- EXIF 존재 여부(has_exif) + 기기 유형(device_category) 추출
- 북마크/저장 기능
- 비로그인 참여 시 로그인 유도 모달

**Step 7: Glossary + Dietary Filter**
- Ingredient Glossary 페이지 (사진, 다국어 설명, 대체재 정보)
- Dietary Filter (Soft Filter 라벨 표시)
- 시드 레시피 +10개, 리믹스 본보기 +2~3개

→ *MVP-β 완료 시점에서 초기 유저 10~20명 인터뷰: "리믹스를 해본 적 있나요?", "레시피 변형 과정을 기록하고 싶나요?" — 트리 시각화 투자 전 수요 검증.*

**Step 11: 쇼핑 리스트 + 연쇄 추천**

- 식재료 연쇄 추천 — 레시피 하단 + 쿠킹 모드 완료 화면
- 냉장고 털이 — 보유 재료 기반 레시피 검색

**Step 12: 온보딩 + 필터 고도화 + Recipe Linter**
- 가입 직후 선택적 온보딩 (비건/할랄/난이도)
- Dietary Filter UX 고도화 (소프트/하드 토글)
- Recipe Linter (재료-스텝 정합성 검사, 단위 누락 감지)

## 14. 현실 체크

1. **콜드 스타트가 가장 큰 리스크** — 기술도 아니고 법도 아님. 시드 콘텐츠 퀄리티 + 영어 자연스러움이 전체 전략의 성패를 좌우함.
2. **닭과 달걀** — 사람이 없으면 콘텐츠 X, 콘텐츠 없으면 사람 X. 초기에는 "큐레이션 사이트" 느낌으로 포지셔닝, "커뮤니티"는 유저가 쌓이면 자연스럽게 전환.
3. **영어 UX 1순위** — 타겟이 외국인인데 한국어 기준으로 설계하면 런칭 후 전부 뜯어고쳐야 함. 영어 기본, 한국어 보조.
4. **MVP 범위 과다 경계** — 기획서가 탄탄한 만큼 "다 넣고 싶은" 유혹이 강함. MVP-α에서 검증할 건 딱 하나: "외국인이 한식 리믹스 레시피를 올리고 싶어하는가?"
5. **Fork & Remix 수요 미검증** — 핵심 차별점이지만 일반 유저에게 "리믹스 트리"가 직관적일지는 미지수. MVP-β 후 유저 인터뷰로 검증 필수. 트리 시각화는 수요 확인 후에만 투자.
6. **수익화까지 긴 공백기** — MAU 수천 달성에 1~2년 이상 걸릴 가능성. 월 $45~80이니 자비 운영 가능하지만, 동기 유지 전략 필요 (포트폴리오 가치, 기술 성장 등).
7. **AI 레시피 서비스와의 경쟁** — ChatGPT에게 물으면 즉시 답이 나오는 시대. "실제로 만들어본 사람의 검증된 결과물"이라는 차별점을 시각적으로 강하게 밀어야 함.
8. **다국어 재료 엔진 범위 관리** — 핵심 30개를 풀 데이터로, 나머지는 점진적 확장이 현실적.
9. **승인권 시스템 복잡도** — 직계 부모 승인권, Master's Choice 등은 V2로 미루는 게 맞음.
10. **글로벌 활동 표시 초기 공허함** — 데이터 적을 때 히트맵 대신 "최근 활동 국가 카드"로 시작.

## 15. 팀 구성 및 역할 분배

### 15.1 인원별 시나리오

**1명 — 올인 개발자**
- 가장 큰 병목: 시드 콘텐츠 제작. 코딩 + 요리 + 촬영 + 영어 작성 동시는 번아웃 직행.
- 시드 콘텐츠는 기존 공개 레시피를 참고해서 본인 버전으로 재작성 (저작권 문제없음).
- 마케팅은 Reddit 위주로 최소한만. TikTok/Reels 포기.
- 타임라인: 13.6 기준의 **1.5~2배**.

**2명 — 가장 이상적 ⭐**

| 역할 | 담당 | 비중 |
|------|------|------|
| Dev + 기술 | A | 풀타임 개발, 인프라, DB |
| 콘텐츠 + 커뮤니티 | B | 시드 레시피 제작, 영어 품질 관리, Reddit/SNS 운영, 유저 피드백 수집 |

이 조합이 가장 현실적인 이유: "닭과 달걀" 문제를 동시에 공략 가능. A가 플랫폼을 만드는 동안 B가 콘텐츠를 쌓고, 런칭 시점에 둘 다 준비된 상태.

B에게 필요한 역량: 영어 네이티브 수준 작문, 요리 가능, SNS 감각. 해외 거주 한국인이 최적.

**3명**

| 역할 | 담당 |
|------|------|
| 풀스택 Dev | A — 핵심 기능 개발 |
| 프론트엔드/UX | B — 쿠킹 모드, 반응형 UI, 모바일 UX 고도화 |
| 콘텐츠 + 마케팅 | C — 시드 레시피, SNS, 커뮤니티 |

3명이면 쿠킹 모드를 V1 초반에 당겨올 수 있음. 다만 3명이 확정이 아니면 2명 기준으로 계획하는 게 안전.

### 15.2 시드 콘텐츠 제작 일정

시드 콘텐츠는 개발과 **별도 트랙으로 병행** 진행. 코딩 끝나고 만드는 게 아니라 동시에 제작.

| 기간 | 개발 트랙 | 콘텐츠 트랙 |
|------|----------|------------|
| 1~2주 | 인프라 세팅 | 시드 레시피 기획, 재료 시드 30개 정리, Reddit 계정 활동 시작 |
| 3~6주 | MVP-α 개발 | 시드 레시피 10개 제작 (요리+촬영+영어 작성), Reddit 활동 |
| 7~10주 | MVP-β 개발 | 시드 +10개, 리믹스 본보기 3개, Reddit 런칭 글 |

## 16. 엣지 케이스 처리

### 16.1 레시피 삭제

- **리믹스가 달린 원본 삭제:** 실제 삭제(hard delete)가 아닌 **소프트 삭제** 처리. 레시피 본문은 비공개되지만, 트리 구조상 노드는 "[삭제된 레시피]"로 유지. 리믹스들의 parent 링크가 끊어지지 않도록.
- **리믹스가 없는 레시피:** 완전 삭제 가능
- **삭제 시 경고:** "이 레시피에 N개의 리믹스가 있습니다. 삭제하면 원본 정보가 비공개됩니다."

### 16.2 유저 탈퇴

| 데이터 | 처리 방식 |
|--------|----------|
| 레시피 | 소프트 삭제 (트리 구조 유지, 작성자명 "탈퇴한 사용자"로 변경) |
| 리뷰 | 익명화 ("탈퇴한 사용자"로 표시, 평점 데이터는 유지) |
| 북마크 / 쇼핑리스트 | 완전 삭제 |
| 프로필 / preferences | 완전 삭제 |
| Glossary 편집 이력 | 편집 내용 유지, 기여자명 익명화 |

### 16.3 리믹스 승인 후 원본 수정

- 원본이 대폭 수정되어도 리믹스는 "승인 당시 버전" 기준으로 유지
- 원본 수정 시 리믹스 작성자에게 알림: "원본이 업데이트되었습니다"

### 16.4 중복 리믹스

- 같은 원본에서 같은 유저가 여러 리믹스 생성 가능 (제한 없음)
- 단, pHash로 사진이 동일하면 경고

## 17. 커뮤니티 가이드라인 & 모더레이션

### 17.1 커뮤니티 가이드라인

**올려도 되는 것:**
- 직접 만든 레시피 (오리지널/리믹스)
- 직접 촬영한 사진
- 출처를 밝힌 참고 레시피 (Inspired by)
- 실패 후기도 환영 ("이렇게 하면 안 됩니다" 류)

**올리면 안 되는 것:**
- 타인의 사진/영상 무단 도용
- 타인의 설명 텍스트 그대로 복사
- 음식과 무관한 콘텐츠
- 광고/스팸성 게시물
- 혐오/차별 발언
- 건강에 위험한 조리법 (예: 생닭 세비체 등 식중독 위험)

**리뷰 가이드라인:**
- "I made this!" 리뷰는 반드시 직접 만든 결과물 기반
- 만들지 않고 평점만 남기는 행위 지양 (사진 필수는 아니지만 권장)
- 인신공격/비하 금지, 레시피에 대한 건설적 피드백 권장

**리믹스 예절:**
- 원본과 뭘 바꿨는지 변경점 설명 권장
- Inspired by 명시 권장 (의무는 아님)
- 원본 작성자에 대한 기본적 예의

### 17.2 콘텐츠 모더레이션 (1인 운영 현실 기반 3계층)

**1계층 — 자동 필터 (즉시):**
- pHash 중복 사진 감지 → 업로드 시 경고
- 금칙어 필터 (욕설, 스팸 키워드)
- 동일 IP에서 단시간 대량 등록 → 속도 제한

**2계층 — 커뮤니티 자정 (수시):**
- 유저 신고 버튼 (사유 선택: 사진 도용 / 스팸 / 부적절 내용 / 기타)
- 신고 N건 누적 시 자동 비공개 처리 + 운영자 알림
- 고등급 유저의 신고에 가중치 부여

**3계층 — 운영자 검토 (비동기):**
- 자동 비공개된 콘텐츠 운영자 큐에 쌓임
- 주 1~2회 검토 → 복원 또는 확정 삭제
- DMCA 신고 접수 시 → 24~48시간 내 조치 (법적 의무)

**처벌 단계:**
- 1차: 해당 콘텐츠 삭제 + 경고
- 2차: 7일 등록 제한
- 3차: 계정 정지

## 20. 개인정보 처리

### 20.1 EXIF 데이터

- EXIF에는 GPS 좌표, 기기 정보, 촬영 일시 등 민감 정보 포함
- 신뢰 점수 계산 후 **원본 EXIF는 즉시 삭제** (서버에 저장하지 않음)
- **저장하는 것 (최소화):**
  - `has_exif: boolean` — EXIF 존재 여부만
  - `device_category: "mobile" | "dslr" | "unknown"` — 기기 유형만 (모델명 저장 X)
- **저장하지 않는 것:** GPS 좌표, 기기 모델명, 촬영 일시 — GDPR 개인정보 리스크 원천 차단

### 20.2 수집 정보

| 항목 | 수집 방식 | 필수 여부 |
|------|----------|----------|
| 이메일 | 소셜 로그인 | 필수 |
| 프로필 사진 | 소셜 로그인 | 선택 |
| 식단 선호도 | 온보딩 | 선택 |
| 국가 정보 | IP 기반 자동 추정 + 유저 수정 가능 | 선택 |

### 20.3 글로벌 법적 고려

| 법률 | 대상 | 핵심 의무 |
|------|------|----------|
| **GDPR** (유럽) | EU 유저 유입 시 | 쿠키 동의 배너, 데이터 삭제 요청 처리 (Right to be Forgotten), Privacy Policy 필수 |
| **CCPA** (미국 캘리포니아) | 캘리포니아 유저 | 개인정보 판매 금지 고지 |

## 21. 이미지 최적화 전략

### 21.1 업로드 시 처리 파이프라인

```
원본 업로드
  → 리사이즈 + 압축 (Sharp 또는 Supabase Edge Functions)
    → 원본 보관용: 최대 1920px (장변 기준)
    → 목록 썸네일: 400px
    → 카드 미리보기: 800px
  → WebP 변환 (용량 50~70% 절감)
  → pHash 생성 + EXIF 추출 → DB 저장
  → 원본 EXIF 삭제 후 저장
```

### 21.2 제공 시

- Vercel Image Optimization 또는 Next.js `<Image>` 컴포넌트 활용
- lazy loading + 반응형 srcset (기기 해상도에 맞는 크기 자동 선택)
- 쿠킹 모드에서는 스텝 사진을 미리 프리로드 (다음 2~3스텝)

### 21.3 용량 제한

| 항목 | 제한 |
|------|------|
| 레시피당 사진 | 최대 10장 (스텝별 1장 + 완성 사진) |
| 리뷰 사진 | 최대 3장 |
| 파일 크기 | 원본 10MB 이하 |
| 압축 후 목표 | 사진 1장당 평균 100~200KB |

이렇게 하면 Supabase 무료 1GB로 레시피 500개+ 커버 가능.

### 21.4 CDN (V1 이후)

- 초기: Vercel Edge Network로 충분
- 트래픽 증가 시: Cloudflare R2 (Supabase Storage 대체, egress 무료) 검토
