# 맛담 (MatDam) — 아이디어 개요

> "맛을 담다"
> 한식 중심의 글로벌 레시피 커뮤니티 플랫폼

---

## 한 줄 요약

**외국인이 한식 레시피를 자기 방식으로 변형(Remix)해서 공유하는 글로벌 플랫폼.**

---

## 왜 이걸 만드려고 하는가

### K-Food 시장이 폭발하고 있다

- TikTok #koreanfood — 월 23억+ 뷰
- 미국 내 한국 식품 시장 전년 대비 28.6% 성장 (2025 Q2)
- K-드라마, K-팝이 한식에 대한 관심을 글로벌하게 견인 중
- 불고기 타코, 김치 토핑 피자 등 퓨전 요리가 전 세계에서 자생적으로 발생

### 그런데 "판"이 없다

한식 레시피를 찾으려면 Maangchi 유튜브, 개인 블로그 등을 전전해야 함.
전부 **일방향** — 보기만 하고, 내 버전을 올릴 곳이 없음.

| 플랫폼 | 형태 | 한계 |
|--------|------|------|
| Maangchi | 1인 유튜브+블로그 | 일방향, 사용자 참여 불가 |
| Korean Bapsang | 1인 블로그 | 커뮤니티 없음 |
| 만개의레시피 | 대형 플랫폼 | 한국어 전용, 외국인 대상 아님 |
| Cookpad | 글로벌 커뮤니티 | 한식 특화 아님 |

> **"한식 + 커뮤니티 + 글로벌" 조합의 플랫폼이 아예 없다.**

---

## 핵심 컨셉

### 1. 레시피 리믹스 (Recipe Remix)

원본 한식 레시피를 기반으로 자기만의 버전을 만들어 공유하는 구조.

예시:
```
정통 떡볶이 (원본)
├── 로제 떡볶이 (리믹스)
├── 짜장 떡볶이 (리믹스)
└── Vegan 떡볶이 — 채수 베이스 (리믹스)
```

누가 뭘 참고해서 어떻게 바꿨는지가 자연스럽게 기록됨.
전 세계에서 한식이 어떻게 현지화되는지 보는 재미.

원본 작성자에게는 자기 레시피가 어떻게 퍼져나가는지 볼 수 있고,
리믹스 작성자에게는 참고한 출처가 자동으로 연결되는 구조.

### 2. 외국인을 위한 한식 재료 가이드

한식의 가장 큰 진입 장벽 = **재료를 못 구함**.

- "고추장이 뭔데?" → 사진 + 설명 + 비슷한 거 비교 (된장 vs miso)
- "내 동네에서 못 구하면?" → 대체재 안내
- "어디서 살 수 있어?" → 현지 구매처 정보
- 단위 변환: 큰술/g ↔ tbsp/oz 원클릭 전환

레시피 안에서 재료를 클릭하면 바로 해당 재료의 가이드로 연결.
나중에는 커뮤니티가 직접 대체재 후기를 쌓아서 데이터가 점점 두꺼워지는 구조.

### 3. 쿠킹 모드 (Cooking Mode)

실제로 요리할 때 쓸 수 있는 화면.
스텝별 카드 한 장씩 넘기면서, 타이머 달고, 화면 안 꺼지게.

기존 레시피 사이트는 전부 "읽는 용도"지, **요리하면서 보는 용도**로 설계된 곳이 없음.

- 각 스텝에서 필요한 재료만 하이라이트
- 폰, 태블릿, 데스크톱 각각에 맞는 레이아웃
- 손이 젖어도 쓸 수 있게 큰 버튼, 간단한 제스처

### 4. 식단 필터 (Dietary Filter)

Vegan / Halal / Gluten-free / Nut-free 등 식단별 필터.

한식은 젓갈(fish sauce), 간장(soy = gluten) 등이 기본 재료라서,
식단 제한이 있는 외국인에게는 "이 레시피 먹어도 되나?"가 큰 고민.
필터로 걸러주거나, 해당 식단에 맞는 리믹스 버전으로 안내.

### 5. AI 레시피가 줄 수 없는 것

ChatGPT에게 한식 레시피를 물어보면 즉시 답을 준다.
맛담이 만들려는 건 그것과 다른 것 — **실제로 만들어본 사람들이 사진과 후기를 남긴, 검증된 레시피들의 집합.**

- "I made this!" — 직접 만든 인증 사진과 후기가 레시피의 신뢰도를 만듦
- AI 레시피 = 이론적 / 맛담 레시피 = 실전 검증
- 리뷰 수, 리믹스 수, 인증 사진 수가 곧 그 레시피의 신뢰 지표

---

## 타겟 유저

- **Primary:** 한식에 관심 있는 외국인 (영어권 우선)
- **Secondary:** 해외 거주 한국인

플랫폼 기본 언어는 영어. 한국어 병행.

**핵심 설계 기준:** "외국인이 이 화면을 처음 봤을 때 3초 안에 이해하는가?"
한국어 기준으로 만들고 나중에 번역하는 게 아니라, 처음부터 영어 기준으로 설계.

---

## 그 외 구상 중인 기능

- **쇼핑 리스트:** 여러 레시피를 골라서 "이번 주 장보기 리스트" 자동 생성. 재료가 겹치면 합산
- **남은 재료 활용 추천:** 떡볶이 하고 남은 어묵으로 뭘 할 수 있는지 자동 제안
- **리뷰 시스템:** 단순 별점이 아니라 맛 / 난이도 / 정통성 / 아이 적합도 등 축별 평가
- **레시피 품질 검수:** 등록 전에 빠진 재료, 모호한 설명 등을 자동 체크해주는 기능

---

## 확장 비전

한식에서 시작하되, 검증되면 일식 → 중식 → 동남아 → 유럽 등으로 단계적 확장.
"맛을 담다"라는 이름은 한식에 국한되지 않음.

리믹스 시스템이 확장되면, 국가 간 식재료와 조리법이 섞이는 과정 자체가 데이터로 기록됨.
예를 들어 한국의 고추장이 멕시코의 타코와 만나서 어떻게 변형되는지 — 이런 흐름을 추적하고 시각화하는 것이 플랫폼의 궁극적 가치.

---

## 현재 단계

- 기획 진행 중
- 시장 조사 및 수요 검증 단계
- 기술 스택 선정 완료
- 프로토타입 (랜딩 페이지) 제작 완료

## 같이 하면 좋을 역할

이 프로젝트는 혼자서도 만들 수 있지만, 콘텐츠(요리+촬영+영어)와 개발을 동시에 하면 속도가 느려짐.
이 비전에 공감하는 사람과 초기부터 같이 만들고 싶음.

### 2명일 때

초기 업무량 참고:

| 영역 | 초기 부담 |
|------|----------|
| 백엔드 | 중간 (DB, API, 인증, 인프라) |
| 프론트엔드 | 무거움 (쿠킹 모드, 슬라이드 입력, 반응형, 다국어) |
| 시드 콘텐츠 | 무거움 (요리+촬영+영어 작성 × 10~20개) |
| 커뮤니티 | 가벼움 (초기 유저 적음) |

**옵션 A: 풀스택 Dev / 콘텐츠+커뮤니티**

| 역할 | 담당 | 비고 |
|------|------|------|
| 풀스택 Dev | A | 프론트+백엔드 전체, 인프라, DB |
| 콘텐츠 + 커뮤니티 | B | 시드 레시피 제작 (요리+촬영+영어), 커뮤니티 운영 |

- 기술 커뮤니케이션 비용이 없음 (프론트/백 합의 불필요)
- A가 플랫폼을 만드는 동안 B가 콘텐츠를 쌓아서, 런칭 시점에 둘 다 준비된 상태
- B에게 이상적인 역량: 영어 네이티브 수준 작문, 요리 가능, SNS 감각

**옵션 B: 프론트+콘텐츠 / 백엔드+커뮤니티**

| 역할 | 담당 | 비고 |
|------|------|------|
| 프론트 + 콘텐츠 | A | UI 개발 + 시드 레시피 제작 |
| 백엔드 + 커뮤니티 | B | DB, API, 인프라 + 커뮤니티 운영 |

- 프론트 담당자가 직접 콘텐츠를 만들면서 자기가 만든 UI를 실전으로 검증 (독 푸딩 효과)
- 단, 프론트(무거움) + 콘텐츠(무거움)로 A쪽 과부하 위험

**옵션 C: 백엔드+콘텐츠 / 프론트+커뮤니티**

| 역할 | 담당 | 비고 |
|------|------|------|
| 백엔드 + 콘텐츠 | A | DB, API, 인프라 + 시드 레시피 제작 |
| 프론트 + 커뮤니티 | B | UI 개발 + 커뮤니티 운영 |

- 백엔드(중간) + 콘텐츠(무거움) / 프론트(무거움) + 커뮤니티(가벼움) = 양쪽 부담이 비슷
- 백엔드 담당자가 콘텐츠를 직접 만들면서 DB 구조를 실전 검증
- 프론트 담당자가 커뮤니티를 보면서 유저 불만을 바로 UI에 반영
- 백엔드 작업은 집중 블록, 콘텐츠는 저녁/주말에 요리하면서 — 리듬이 자연스러움

### 3명일 때

**옵션 A: 풀스택 / 프론트+UX / 콘텐츠+커뮤니티**

| 역할 | 담당 | 비고 |
|------|------|------|
| 풀스택 Dev | A | 핵심 기능 개발, 인프라, DB |
| 프론트엔드 / UX | B | 쿠킹 모드, 모바일 반응형 UI, 사용자 경험 설계 |
| 콘텐츠 + 커뮤니티 | C | 시드 레시피 제작, 커뮤니티 운영 |

- 가장 깔끔한 분리. 각자 영역이 명확
- 쿠킹 모드 같은 UX 집약적 기능을 초반부터 병렬로 개발 가능

**옵션 B: 백엔드 / 프론트+커뮤니티 / 콘텐츠 전담**

| 역할 | 담당 | 비고 |
|------|------|------|
| 백엔드 | A | DB, API, 인프라, 검색 |
| 프론트 + 커뮤니티 | B | UI 개발 + 유저 피드백 수집 → UI 반영 |
| 콘텐츠 전담 | C | 시드 레시피 제작에만 집중 (요리+촬영+영어) |

- 콘텐츠 전담자가 있으면 시드 레시피 퀄리티가 확 올라감
- C가 개발자가 아니어도 됨 — 요리+촬영+영어만 잘하면 최적
- 프론트 담당자가 커뮤니티를 보면서 유저 반응을 바로 UI로 반영

**옵션 C: 백엔드+콘텐츠 / 프론트+UX / 커뮤니티+콘텐츠 보조**

| 역할 | 담당 | 비고 |
|------|------|------|
| 백엔드 + 콘텐츠 리드 | A | DB, API + 레시피 기획/요리 |
| 프론트 + UX | B | UI 개발, 쿠킹 모드, 반응형 |
| 커뮤니티 + 콘텐츠 보조 | C | 커뮤니티 운영 + 영어 교정/촬영 보조/SNS |

- 콘텐츠 작업을 A와 C가 나눠서 부담 분산
- A가 요리하고 C가 영어 교정+SNS 포스팅 — 콘텐츠 파이프라인 형성
- 아무도 과부하 걸리지 않는 가장 균형잡힌 구조

### 참고

3명이 확정이 아니면 2명 기준으로 계획하는 게 안전.
어떤 조합이든 핵심은 **개발과 콘텐츠가 동시에 진행되는 것**.
